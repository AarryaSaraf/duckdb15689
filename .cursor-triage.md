# TPC-DS Cardinality Feedback Experiment Triage

## 1. Experiment Overview
The goal is to implement an automated, iterative cardinality feedback loop for TPC-DS queries. By capturing the **actual** join cardinalities from a query's physical plan and injecting them back into the optimizer as **estimates**, we aim to achieve plan convergence. This can be done since TPCDS queries do not modify the data 
and we do it for one query at a time. 

### Key Components:
- **Modified DuckDB (C++)**: 
    - Logs every logical join estimation to `cardinality_log.txt`.
    - Loads pre-computed cardinalities from `actual_cardinality.json` at startup.
    - Injects these values into the `CardinalityEstimator` during query planning.
- **Feedback Script (`feedback_benchmark.py`)**: 
    - Executes queries with `PRAGMA enable_profiling = 'json'`.
    - Parses the physical plan and cardinality logs.
    - Matches physical join nodes to the logical expressions logged by the optimizer.
    - Updates the injection JSON and repeats the process.

## 2. Technical Modifications
We implemented several robust mechanisms to ensure accurate matching between the physical plan and the optimizer's logical expressions:

- **Condition Normalization**: Join conditions are normalized (e.g., `A=B` is equivalent to `B=A`) and split across multi-way joins to allow stable matching.
- **Descendant Table Context**: Each physical join node is tagged with the set of tables scanned in its subtree (`SEQ_SCAN` nodes). Matching requires that the physical join's tables are a subset of the logical expression's table set.
- **Two-Phase "Estimated Cardinality" Matching**:
    - **Phase 1 (Precision)**: For already-injected expressions, the script matches by finding the physical node whose `Estimated Cardinality` exactly equals the injected value. This provides a 1:1 map that is immune to condition rewrites.
    - **Phase 2 (Fallback)**: For new expressions, it falls back to normalized condition and table-set matching.
- **Verification Suite**: A 4check verification framework runs on every iteration to ensure that (1) DuckDB loaded the JSON, (2) all expressions were logged, (3) all non-CTE joins were injected, and (4) for converged plans, Injected == Actual.

## 4. C++ Modification Details
All C++ side changes are contained within **`src/optimizer/join_order/cardinality_estimator.cpp`** and 
** `duckdb15689/src/optimizer/join_order/relation_statistics_helper.cpp` **.

### Key Logic & Changes:
1.  **File-Path Constants**: Added paths for `actual_cardinality.json` (injection source) and `cardinality_log.txt` (logging sink).
2.  **`load_cardinality_data()`**: 
    - Uses the `nlohmann/json` library to parse the injection JSON.
    - Loads data into a persistent `unordered_map<string, double>` for fast lookup during the optimization phase.
3.  **`EstimateCardinalityWithSet<double>` (The Core Overload)**:
    - **Key Construction**: 
        - Iterates through the `JoinRelationSet` to extract table names from the `CardinalityHelper` objects.
        - Gathers all join filters (`FilterInfo`) associated with the current relation set.
        - Formats these into the `LOGICAL_JOIN: RelSets: [...] Tables: [...] Filters: [...]` string key.
    - **Lookup & Injection**:
        - Checks the loaded map for this exact string key.
        - If found: Sets the result to the injected value and logs the event (e.g., `... using INJECTED Cardinality: X`).
        - If not found: Proceed with standard DuckDB cardinality estimation and logs the estimate (e.g., `... Estimated Cardinality: Y`).
4.  **Table Name Tracking**: Modified `CardinalityHelper` to store `table_names_joined` (as a `vector<string>`), which is populated by `InitCardinalityEstimatorProps` when base table stats are loaded. This allows propagating table names up the join tree for string-key construction.

## 3. The Core Bug: Semantic Context Mismatch
During testing (specifically on **Q23**), we identified a fundamental limitation in the current C++ injection logic.

### Mismatch Details (Q23 Example):
We injected a cardinality of **12,537** for a logical join involving `{catalog_sales, customer, date_dim}`. 
- **The Expression Key**: `LOGICAL_JOIN: RelSets: [0, 1, 2] Tables: [catalog_sales, customer, date_dim] Filters: [(cs_bill_customer_sk = c_customer_sk) AND (cs_sold_date_sk = d_date_sk)]`
- **Expected result**: The optimizer applies 12,537 to the "inner" join in the subquery.
- **Observed result**: The optimizer applied the 12,537 estimate to an **outer join** between the same tables that used the same filters but represented a different part of the plan (one that actually saw 130k rows).

## 6. Reproducing the Bug & Log Locations

### Step-by-Step Reproduction (Q23 Example):
1.  **Configure the Benchmark**: Open `feedback_benchmark.py` and set `TARGET_QUERIES = [23]`.
2.  **Clear Previous Data**: Delete `actual_cardinality.json` and `cardinality_log.txt` to start fresh.
3.  **Run the Loop**: Execute `/usr/local/bin/python3 feedback_benchmark.py`.
4.  **Observe Iteration 2**:
    - Look at the `cardinality_log.txt`. You should see a line stating: `LOGICAL_JOIN: ... Tables: [catalog_sales, customer, date_dim] ... using INJECTED Cardinality: 12537.000000`.
    - Compare this to the Physical Plan output in the terminal. Locate the Join node that has **`Estimated Cardinality: 12537`**.
    - Observe its **`Actual Cardinality`** is **~130,972**.
    - Notice that this node's join condition is usually simpler (e.g., `c_customer_sk = cs_bill_customer_sk`) than the filter set in the logical expression key.

### Log and Data Locations:
All paths below are relative to `duckdb15689/`:

- **`cardinality_log.txt`**: The **primary trace file**. Every time the DuckDB optimizer estimates a join cardinality, it logs the logical expression key and whether it was an "Estimated" or "INJECTED" value.
- **`actual_cardinality.json`**: The **injection database**. This file is generated by the Python script and read by DuckDB at the start of every query optimization. It maps expression keys to cardinality values.
- **`feedback_benchmark.py`**: The **orchestration script**. It handles the iterative runs, parsing, and the matching logic that reveals these C++ side mismatches.
- **`src/optimizer/join_order/cardinality_estimator.cpp`**: The **source of truth**. This is where the C++ side lookup and injection logic resides.

## 7. Current Status
- **Successes**: Q1, Q2, Q3, Q4, and Q5 now all **converge** perfectly in 2 iterations thanks to the table-aware matching.
- **Open Issues**: Q23 and others oscillate due to the C++ context mismatch described above. 
- **Recommendation**: To fix this, the C++ side lookup key must be made more specific, perhaps by including parent-child relationships or more granular identifiers from the logical plan.
